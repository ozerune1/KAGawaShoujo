Title:  Basic Usage of World Extension

Author: Wamsoft L.P.



● Overview



This section explains the basic procedures for scripting using the 
World Extension features in "Kirikiri/KAGEX".



● Preparation



Configure the folder structure for Kirikiri/KAGEX.



 game/

  krkr.exe				Kirikiri Executable

  plugin/				Plugin files

	extrans.dll			Extended transitions

	krmovie.dll			DLL for Movies

	wuvorbis.dll		DLL for OGG playback

	csvParser.dll		CSV parser extension

	layerExImage.dll	Image function extension

  data/

	system/				Copy the distribution from Kirikiri KAGEX

	main/				Scripts for the system.

						It is recommended to copy the KAGEX sample

                        and start working from there.



Executing krkr.exe will start the system.



When KAGEX starts, it first attempts to execute first.ks.

The sample system is set up to start in the following order:



(1) Initialization processing in first.ks

(2) Launch title.ks at the end of first.ks

(3) Launch start.ks when "Start from Beginning" is selected in title.ks



If you want to start from an arbitrary scenario file, open "Load Scenario" 
from the debug menu and enter the scenario filename there. 
It will start from the beginning of that scenario.



● Basic Structure of a Scenario File



Scenario files are placed in data/scenario.



# Actually, they can be anywhere, but

# keeping them in separate folders makes them easier to understand.



-----------------------------------------------------

; ← This description is a comment

; ↓ This description becomes a label. Make it unique within the scenario file.

*label|Title

; Anything after | becomes the title for saved data.

; If you want to inherit the title, specifying *label| (empty) is OK.



; ↓ Initialization of scene state.

[initscene]

; The content is defined in macro.ks, and basically includes:

; (1) Execution of World Extension's [init]. Initializes all character states, etc.

; (2) Message window initialization processing.

; (3) Other necessary processing.



【Name】Conversation text

 .....

 .....



[jump storage=Next Scenario File]

-----------------------------------------------------



● Concrete Description Examples



Below are examples of descriptions for specific situations.



Legend

　Background: School, Hallway

　Name: Akari



◇ I want to display a background



 (1) Prepare the files



　 Create data for different time zones such as Morning, Day, and Night.

　 Please unify the file naming rules throughout.



　Example

   bg01_0  School Morning

   bg01_1  School Day

   bg01_2  School Night



   bg02_0  Hallway Morning

   bg02_1  Hallway Day

   bg02_2  Hallway Night



　 Place files in data/bgimage.



 (2) Register background information in envinit.tjs



　> Refer to world_inst.txt for details.



　・Reference name

　・Image file (with time variations)



  ※ Time specifications are also registered at the same time.



 (3) Use the registered name in the script to display it.



 --------------------------------------------

 [School Day]  ; Display with time specification

 ....

 [Classroom]   ; Change only the place to Classroom, keeping the time as is

 ....

 ---------------------------------------------



◇ I want to display a character



 (1) Prepare the files



   Place files in data/fgimage.



   Typically:



   akariX_AAA_BBB_CCC.png



   X  : Level specification

   AAA: Pose specification

   BBB: Dress specification

   CCC: Face specification

　　

　 Construct the filename by expanding combinations for each element in this form.

 　※ Elements can be numbers or strings.

 

   ※ You do not need to prepare every possible combination.

　　 Only what is necessary is OK.



 (2) Register character information in envinit.tjs



　　> Refer to world_inst.txt for details.



　　The form includes elements such as:

　　・Pose

　　・Dress

　　・Face

　　So register those names.

　　

　　Concept:

　　　Pose: Same pattern of gestures/body language

　　　Dress: What they are wearing

　　　Face: Facial expression



　　If there are no issues in terms of the game, it is fine not to use "Dress",

　　or not to use "Pose" (fixed).



　　Regarding size, the element:

　　・Level

　　exists.



    This "Level" is a concept for moving back and forth within the character screen.



　　Small < Back, Front, Foreground > Large

　　Generally, create and use 2 to 3 stages.



　　※ If using levels, you must register level specifications 

　　　 in envinit.tjs in advance.



　　Standing image data undergoes color correction linked to the "Time" 

　　specified in the background registration.



 (3) In the script, describe the registered name and some pose, 

　　 and it will be displayed on the screen.



 ----------------------------------

  [Akari Uniform]

 ----------------------------------



◇ I want to specify the display position of a character



(1) Register the character



(2) Register keywords for display positions in envinit.tjs



　Left/Right Position

　　Left, Center, Right, etc.



　Front/Back Position

　　Back, Front, Foreground, etc.



　Special Display Positions

　　Enter, Exit, Face, etc.



  Relative Movement Specification

　　MoveLeft, MoveRight, etc.



　※ Keywords can be set arbitrarily. The sample includes the examples above.

 

(3) Specifying the location



 ----------------------------------

 [Akari Left]

 ----------------------------------



 With the tag specifying the character name, list the location name and it will be displayed there.



(4) Moving location



　If you specify a location for a character already displayed,

　and it is different from the current location, they will move there and be displayed.

　At this time, dress and pose are inherited as is.



 ---------------------------------------------

  [Akari Uniform Left]     ; Appears on the left

　【Akari】Good morning.



　[Akari Right]            ; Moves to the right

  【Akari】It's a nice day today too.

 ---------------------------------------------



(5) Moving location (Relative specification)



　Relative movement can also be used.



 ---------------------------------------------

  [Akari Left Uniform]     ; Screen left

  [Shiori Center Uniform]  ; Screen center

  ....



  [Akari MoveRight Uniform]  ; Moves right to come to the center position

  [Shiori MoveRight Uniform] ; Moves right to come to the right position

 ---------------------------------------------



◇ I want to display an Event CG



(1) Prepare the image



　Prepare the image for the Event CG.



　Place it in data/bgimage.



(2) Display



  [ev filename]



　Simply specify the filename to display.



(3) Erase



 To erase explicitly:



　[ev hide]

  

 Auto-erase processing:



　If a background display is performed, the Event CG is automatically erased.



  Example

  -------------------------------

  [ev ev001]

  .....

  [School Night]  ; If a background is specified, the event CG is automatically erased

  -------------------------------



◇ Omitting commands



 In KAGEX World Extension, there are two types of commands.



 Basic 1

  [CommandName Parameter=Value  Parameter=Value]



 Basic 2

  [Name Parameter=Value  Parameter=Value]



 Point 1

  Command types cannot be distinguished by simple appearance.

　The Basic 2 form is a valid style for things that are "registered".

　Example: Character names, etc.



 Point 2

  When using registered "Values", the "Parameter" part can mostly be omitted.



　Example:

　　[Akari pos=Left]        →Omitted→ [Akari Left]

    [Akari pose=Proud]      →Omitted→ [Akari Proud]



　Precaution: When registering "Values", it is assumed you use names that do not duplicate others.



　Example:

　　If you registered "Normal" for Dress and "Normal" for Face:



　[Akari dress=Normal face=Normal]  ; Works as expected

  [Akari Normal]                    ; Even if intended as a face specification, "Normal"

                                    ; might be processed as a Dress!!!

  

　Although there is a priority order when duplicate specifications occur, it is "undefined behavior",

　so please do not use it.



◇ Text Display and Voice Playback



※ The explanation here assumes line mode "page".

　For details on line mode, refer to kagex.txt.



Basic Format

-----------------------------------------

【Akari】「Good morning」

-----------------------------------------



Specify the character's name in the "【Akari】" part.





With this name specification, the following occur:



(1) If there is a name field in the message window, the name is displayed there.

(2) If there is a face field in the message window, a face image linked to the current standing picture is displayed there.

(3) Playback of voice.



※ For (2) and (3), information must be set during character registration 

  in envinit.tjs.



Example of Voice Playback



-----------------------------------------

[Akari voice=1] ; Set voice number to 1



【Akari】「Good morning」     ; akari0001 is played



【Akari】「Hello」            ; akari0002 is played



【Akari】「Goodbye」          ; akari0003 is played

-----------------------------------------



For each character name, the number automatically counts up

and the voice plays.



If you want to change the name notation on the screen:



　Example: The name is not known yet, so I want to display "???"



------------------------------------------------------------

【Akari/???】「It's morning~」 ; The specified voice plays

------------------------------------------------------------



The voice specification is for "Akari", but the display becomes "???".



When specifying voice individually, specify the voice command one by one.



-----------------------------------------

[Akari voice=1] ; Set voice number to 1

【Akari】「Good morning」     ; akari0001 is played



[Akari voice=10] ; Set voice number to 10

【Akari】「Hello」            ; akari0010 is played



[Akari voice=akari0004x] ; Direct filename specification

【Akari】「Goodbye」          ; akari0004x is played



【Akari】「Ahhhh」            ; akari0011 is played

-----------------------------------------



If you specify something other than a number (mixed with alphabet), you can specify

a filename for a single shot. The numeric count carries over to the next one.



※ Warning!!!

  Kirikiri treats numbers starting with 0 as octal numbers.

  Specifying "0010" is not 10, but numerically 8.



◇ Playing/Stopping BGM



You can play BGM.



 (1) Playback



 [bgm MUS001]



 However, if the name starts with "BGM":



 [BGMXXX]



 It plays just with that.



 (2) Stop



 [bgm stop]



 (3) When you want to fade



 Fade In



 [bgm MUS001 time=1000] 



 Fade Out



 [bgm stop=1000]



 (4) When you want to wait for the fade process



 [bgm stop=1000 wait]



◇ Playing/Stopping SE (Sound Effects)



 (1) Play one-shot



 [se SOUND001]



 However, if the filename starts with "SE":



 [SE00001]



 It plays just with that.

 This description remains valid even with the following command specifications.



 (2) Play in loop



 [se SOUND01 loop]



 (3) Stop playing SE (both loop and non-loop)



 [se SOUND01 stop]



  Basically, you need to specify the SE directly.



 (4) Stop all playing SEs



 [allse stop]



 (5) When you want to fade



 Fade In



 [se SOUND01 time=1000]



 Fade Out



 [se SOUND01 stop=1000]



 (6) Want to wait for fade processing



 [se SOUND01 stop=1000 wait]



◇ Advanced Movement



(1) Slide Movement



 If there is a `time` parameter during movement, it will slide move.



 ---------------------------------------------

  [Akari Left Uniform]     ; Appears on left

　【Akari】Good morning



　[Akari Right time=1000]  ; Moves smoothly to the right

  【Akari】It's a nice day today too.

 ---------------------------------------------



(2) Specification of Initial Position for Movement



 In movement specification, if you specify multiple positions separated by ":",

 you can initialize the position instantly just before moving.



 --Format--

   Destination:InitialPosition



 ---------------------------------------------

  [Akari Left Uniform]           ; Appears on left

　【Akari】Good morning



　[Akari Right:Center time=1000] ; Instantly moves to Center, then moves smoothly to Right

  【Akari】It's a nice day today too.

 ---------------------------------------------



◇ Layer Attributes



(1) Specifying Attributes



　You can specify the following attributes for characters and event CGs placed on the screen.



  rotate   Rotation angle

　zoom     Magnification

　opacity  Opacity



  ---------------------------------------------

  [Akari Uniform]

 【Akari】Mornin'~

  

  [Akari zoom=200]

 【Akari】Gigantification~

  ---------------------------------------------



  When specifying for background:       [env zoom=200]

　When specifying for Event CG:         [ev zoom=200]



(2) Slide Movement of Attributes



 These attribute parameters can also be changed smoothly by specifying `time`,

 just like location specification for movement.



  ---------------------------------------------

  [Akari Uniform]

 【Akari】Mornin'~

  

  [Akari zoom=200 time=1000]

 【Akari】Smooth gigantification~

  ---------------------------------------------



(2) Initialization Specification of Attributes [sic]



 Just like movement, if you specify multiple values separated by ":",

 you can initialize the value instantly just before the change.



 --Format--

   TargetValue:InitialValue



  ---------------------------------------------

  [Akari Uniform]

 【Akari】Mornin'~

  

  [Akari zoom=200:80 time=1000]

 【Akari】Zooming from small to big~

  ---------------------------------------------



◇ Screen Switching



What is a Transition...



　Switching smoothly between two different display contents is called

　a "Transition" (Kirikiri terminology).



　Example: Switching character expressions with a crossfade.

　　　Switching the screen with a water-ripple effect.

　Etc.



　Kirikiri comes prepared with many transition libraries by default

　which can be used.



　※ Reference: http://devdoc.kikyou.info/tvp/docs/kr2doc/contents/



In World Extension, transitions are generally used after being "registered" in advance.



(1) Registering Transitions



  Give the transition a name and register it in envinit.tjs.



　Example: normal  Crossfade 300ms for standard character switching



  Please refer to world_inst.txt for details.



(2) Specifying Transitions



 Specifying the `trans` parameter for a character executes the transition

 when the display changes.



 ---------------------------------------------------------

  ; Display with normal expression

  [Akari Uniform Normal]

  【Akari】Mornin'

  

  ; Switch expression to "Laugh". Transition is normal.

  [Akari Laugh trans=normal]

　【Akari】It's a nice day today too.

 ---------------------------------------------------------



　If the registered transition name can be uniquely identified,

　`trans=` notation can be omitted.



  [School trans=normal]  →Omitted→  [School normal] 



  The usage is basically the same for backgrounds and event CGs.



 ---------------------------------------------------------

  ; Display with transition for events

  [ev ev001 trans=event] 



  ....

  

  ; Switch background to school

  [School trans=normal] 

 ---------------------------------------------------------



　※ Note: Registered names can be used for any element.





・Multiple Simultaneous Transitions



　When a transition is specified, it waits for the screen display to complete there,

　so you cannot execute transitions simultaneously for multiple elements

　(Background and Character, Two Characters, etc.).



　Example

 ---------------------------------------------------------

  [Akari Laugh]

  [Hikari Laugh]

  ; In this case, both are instant, so they appear visible simultaneously.



  [Akari Laugh trans=normal]

  [Hikari Laugh trans=normal]

　; Hikari's transition happens after Akari's transition finishes.

 ---------------------------------------------------------



  To do this simultaneously, use the special commands `begintrans` / `endtrans`.



 ---------------------------------------------------------

  [begintrans]

  [Akari Laugh]

  [Hikari Laugh]

  [endtrans trans=normal]

 ---------------------------------------------------------



  With this, both changes are executed simultaneously, and the transition

　is performed on the whole result.



　Example of changing simultaneously with background



 ---------------------------------------------------------

  [Hallway]

  ....



  [begintrans]

  [School]

  [Akari Uniform Laugh]

  [Hikari Uniform Laugh]

  [endtrans trans=normal]

 ---------------------------------------------------------



・Default Transition



In the standard state, transitions are not executed unless explicitly specified.



In actual scripting work, transitions are used in most scenes,

so the amount of writing becomes large.

Therefore, for commands related to display, you can automatically apply

a transition when that command is executed.



Items for which Automatic Transition can be specified:



Character

・Pose change

・Dress change

・Face change



Background

・Location change

・Time change



Event CG

・Image change



Individual settings are possible for these.



 Example: Before Auto-Application

 ---------------------------------------------------------

  [ev ev001 trans=event] 

  ...



  [School trans=stage] 

  [Akari Uniform trans=charpose]

 ---------------------------------------------------------



 Example: After Auto-Application



 Register the following in envinit.tjs

  eventTrans=event;

  stageTrans=stage;

  poseTrans=charpose;



　This results in exactly the same behavior.

 ---------------------------------------------------------

  [ev ev001] 

  ...



  [School] 

  [Akari Uniform]

 ---------------------------------------------------------



Other Precautions



 ・Direct specification with the `trans` command takes priority over automatic transitions.

 ・You can explicitly specify "No transition" with `trans=void`.



Refer to world_inst.txt for details on default transition registration.



◇ Character Actions



What is an Action...



　A function to make images placed on the screen continue to move automatically is called an "Action".

　

　Example: Shaking violently

　　　Bowing motion

　Etc.



　Actions are realized by specifying action commands and parameters.



In World Extension, actions are generally used after being "registered" in advance.



(1) Registering Actions

  

  Give the action a name and register it in envinit.tjs.



　Example: "Bow" - A motion where the character sprite sinks down in a sine curve and moves back.



(2) Specifying Actions



 Specifying the `action` parameter for a character starts the action execution

 at that timing.

 

 ---------------------------------------------------------

  ; Display with normal expression

  [Akari Uniform Normal]

  【Akari】Mornin'

  

  [Akari Laugh action=Bow]

　【Akari】Good day.

 ---------------------------------------------------------



　If the registered action name can be uniquely identified,

　`action=` notation can be omitted.

　

 [Akari action=Bow] →Omitted→ [Akari Bow] 



  The usage is basically the same for backgrounds and event CGs.



 ---------------------------------------------------------

  ; Action for background image

  [env Bow] 

  ; Action for event CG image

  [ev action=Bow] 

 ---------------------------------------------------------



◇ Control of Message Window



Basically, the message window automatically appears when there is text display.



(1) Manually Erase / Display directly



 [msgoff]　Erase message window

 [msgon]   Display message window



 Transition during erase/display can be specified with `trans=` parameter.



(2) Transition Linkage Control



 If you add the parameter "msgoff" when registering transition information in envinit,

 the message window will be erased before that transition starts.



 When creating rules like "Always hide the message window when displaying an event CG",

 it is convenient to register a transition with this parameter specified.



 This function is often used in combination with the `transwait` parameter 

 (function to wait a little after transition).



◇ Selections and Branching



  Selection processing using links via the message window in KAG can be used as is.



   http://devdoc.kikyou.info/tvp/docs/kag3doc/contents/Diverge.html



  In KAGEX, in addition to this, selection processing using "Selection Layers" is available.



 Registering Selections (Multiple times)

  [seladd text=Content of Selection storage=Jump Destination target=Jump Label exp=Execution processing upon selection]



 Executing Selection Processing

  [select]



 After selection processing execution, it transitions, or executes the `exp` processing,

 then proceeds to the next statement.



◇ Variables



　Variable processing uses the functions of Kirikiri/KAG as is.



　http://devdoc.kikyou.info/tvp/docs/kag3doc/contents/Var.html



◇ Cut-in Processing



If you want to display a cut-in image on the screen, you can generate and use a "Layer".



(1) Create and display a layer



 [newlay name=layer0 file=layerimage show]



 Reads the image "layerimage" and generates a layer.

 By default, it is placed centered on the screen.



 The concept of `level` is valid for layers just like characters.

 Setting `level=4` allows display in front of event CGs.



(2) Moving the layer



 You can specify the layer display position with `xpos`, `ypos` parameters.

 You can manipulate it using the name specified during layer generation,

 similar to a character.



 [layer0 xpos=200 ypos=200 time=1000]

 

・Screen center is the origin

・Image center is the image origin



　Please note these points.



(3) Specifying actions for the layer



 This specification can be done just like for characters.



 [layer0 action=Bow]



(4) Erasing the layer



　Simple display control is done with show/hide.

 

  [layer0 hide]



 When the scene using the layer is over, if you just set it to hide, it simply disappears visually,

 but the internal data remains. To discard this completely, use the `hidelay` command.



 [hidelay name=layer0]

◇ Support for Scene Recollection



 Kirikiri/KAGEX has a special "Recollection Mode" prepared for scene recollection.

 When using this mode, it is necessary to embed descriptions like the following in the scenario.



 -----------------------------------------------------------

 ; Insert a label for jumping at the beginning of the recollection part

 *RecollectionStart|

 [if exp='kag.isRecollection']

 ; Write initialization processing for recollection scene here.

 ; It is unnecessary at the beginning of a scene, but if starting from the middle of a scene,

 ; it may be necessary to restore backgrounds, standing sprites, etc.

 [endif]



 Recollection part body



 ; End recollection specification. Ignored during normal execution,

 ; but ends with this specification in recollection mode.

 [endrecollection]

 -----------------------------------------------------------

 

 In KAGEX terms, recollection mode can be started by using `[recollection]`,

 but in many cases, `kag.startRecollection()` is called directly

 due to start control being handled by the system.



● Other Precautions, etc.



◇ File Overlap



　Even in different folders, the same filename causes conflicts.

　Since there is automatic extension processing, be careful as it might be dangerous even if extensions differ.



◇ Registration Names



　There are no specific restrictions on characters for registration names.

　Using Japanese like in the examples of this document makes it visually easy to understand,

　but writing might be tedious, so registering in English words or Romaji might be faster for work.

　Using copy-paste tools is also a good idea.

● Various Applications



◇ How to display a character in front of an Event CG?



 Example: Appearance in the center of the screen



☆ Point

　States such as character location, dress, and face are always maintained.

　As a rule, try to describe "only the differences".



　-> Specify dress etc. generally only at the first appearance in the scene.



☆ Point

　Although various transitions can be used for character appearance,

　specifying a simple `fade` is the basic method.



　Since a transition is effectively executed even for a single character,

　the load is surprisingly high. If you want to make a work with good tempo,

　it is thought better not to use `fade` for expression changes.
