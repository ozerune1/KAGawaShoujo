● Introduction

This document describes how to register various materials such as characters and backgrounds
into envinit.tjs, which is extended in Kirikiri/KAGEX, and basic scripting methods.

● Registration Overview

In Kirikiri/KAGEX, to simplify scripting, you can register materials frequently used in scripts,
such as characters and backgrounds, into a file called envinit.tjs.

For example, normally in KAG, trying to display a background or character (standing sprite) requires:

[image storage="school" page=fore layer=base]
[image layer=0 page=fore storage="akari" visible=true left=340 top=100]

This script is necessary, but in Kirikiri/KAGEX, you can display backgrounds and characters
using a name-designated script like this:

[School]
[Akari Enter Middle]

However, to perform scripting in this method, registration is required in advance in the file envinit.tjs.

envinit.tjs itself consists of TJS2 dictionary information.

● Basic Notation

The concepts forming the basis of the notation are the following two:

・Arrays

An array is a simple enumeration of multiple pieces of information of the same type.
In TJS, arrays can be defined using [].

To reference a value in an array, use the notation: arrayName[indexNumber].

Array notation example
--------------------------------------------------
var a = [ 1, 2, 3, "text", "text2"];

dm(a[0]); // Displays "1"
dm(a[3]); // Displays "text"
--------------------------------------------------

For details, please refer to http:// (Link omitted)

・Dictionaries (Associative Arrays)

A dictionary records combinations of names and values.
In TJS, dictionaries can be defined using %[] .

To reference a value in a dictionary, use the notation: dictionaryName["KeyName"] or dictionaryName.KeyName.

Dictionary notation example
--------------------------------------------------
var dic = %[ "num" => 1, "date" => "2006/1/2" ];  // perl style

dm(a.num);       // Displays "1"
dm(a["date"]);   // Displays "2006/1/2"

// If the key is a string, the following notation can also be used
var dic = %[ num:1, data:"2006/1/2" ];
--------------------------------------------------

For details, please refer to http:// (Link omitted)

You must enclose the definition with %[ ].

● Creation and Registration of Background Image Data

Normally, create at screen size (800x600).

Create variations for each time of day (Morning/Day/Night), and register the information
in `times` and `stages` in envinit.tjs accordingly.

Example
-------------------
  bg01_0  Home Morning
  bg01_1　Home Day
  bg01_2　Home Night
  bg02_0  School Morning
  bg02_1  School Day
  bg02_2  School Night
  ....
-------------------

Registration example in envinit.tjs
--------------------------------
    "times" => %[
        "Morning" => %[
            "prefix" => "0",
            ],
        "Day" => %[
            "prefix" => "1",
            ],
        "Night" => %[
            "prefix" => "2",
            ],
		],

    "defaultTime" => "Day",
	
    "stages" => %[
        "Home" => %[
            "image" => "BG01_TIME",
            ],
        "School" => %[
            "image" => "BG02_TIME",
            ],
		],
--------------------------------

If you specify a large background for directing purposes,
it is convenient to add the initial coordinate position to the `stages` description.

● Creation and Registration of Standing Character Data

◇ Concept of Standing Character Data

(0) Level
　Prepare separate data based on the front-back position.

(1) Pose
　Those where the basic silhouette is the same.
　Those where expression patterns can basically be reused.

(2) Dress (Clothing)
　Variations of clothing for a certain pose.

(3) Face (Expression)
　Variations of expression for a certain pose.

◇ Perspective (Depth) Level Processing

When using perspective processing, add items for level specification.
Prepare all images scaled for each level, and add the level specification part to the filename.

◇ Setting Method

  1. Assign numbers starting from 0, sequentially from the back-most part.
  2. Configure `levels` in envinit.tjs.
    Parameter `zoom`
　　　Specify what percentage size the image of the specified level is relative to 
      "Standard: Position where movement amount equals pixel value". Used for coordinate correction.
　　Parameter `name`
      Specify the name to insert into the LEVEL part of the image filename specification
      (imageName or faceImageName).

  3. Register names matching the target levels in `positions`.

# Example case
# Register as 0:Back, 1:Front, 2:Foreground, and specify 1 as the default (defaultLevel).
# However, the filenames are adjusted according to the previous registration as follows:
# Back: hoge_1_X_X.png  (60% size of original image)
# Front: hoge_0_X_X.png (Normal image)
# Front: hoge_2_X_X.png (120% size of original image) - *Note: The source text says "Front" for both 0 and 2, but context implies 2 is Foreground/Close-up.*

◇ Expression Composition Methods

Two variations can be used for creating data.

A. Expression Composition Method

 Base Image: Combination of (1) x (2)
 Expression Image: Prepare only the diffs (3) to be composited onto the expression image, for each type of (1).

B. Standalone Method

 Create all patterns of (1) x (2) x (3).

◇ Setting Method

1. Expression Composition Method

Works with this logic if the parameter `faceImageName` is defined.

 1. Detect the base image by the combination of Pose and Dress.
 2. Composite the expression part image onto that image.
 3. Display the composited image as the standing character.

 Expression files must be prepared so they do not misalign for each pose.
 Expression files are reused within the same pose.

2. Expression Integration Method

Works with this logic if the parameter `faceImageName` is not defined.

In this case, since there is no need to composite expression images, there is no problem
even if the shape is completely different between Poses/Dresses, so there is no need to use
either "Pose" or "Dress".

1. "Pose" is unspecified (only one), handling switching with only "Dress" and "Expression".

　Include both DRESS and FACE in the Pose parameter.

2. "Dress" is unspecified, handling switching with "Pose" and "Expression".

　Include only FACE in the Pose parameter.

◇ Regarding Creation of Standing Character Data

The origin position of standing character data is "Bottom Center" of the image.

Notes during data creation:

(1) Align the "feet" position for all characters.

(2) Split into individual characters.

　　☆ Do not trim the bottom edge of the image (because the origin will shift).
　　☆ You may arbitrarily trim the top edge of the screen.

(3) Left/Right position adjustment of standing pictures per character.

　　Align the center position for all standing picture variations of a certain character.
　　After this, for individual poses, you may perform left/right trimming,
　　but you must be careful not to shift the center (need to trim the same amount from left and right).

(4) Create the expression layer at the same size as the base layer.
　
　　This is to properly align the coordinates of the expression.
　　In Photoshop, using "Save for Web" allows outputting as PNG while keeping the same size as the base layer.

  　In the case of PNG format, if `off_x` and `off_y` can be properly preserved,
    you may trim it.
	Example: Trimming using imagemagick
    > mogrify -trim *.png


● Detailed Explanation of characters

First, the structure of a character's standing picture is as follows:

　・Basic Pose (filename)
　・Depth (LEVEL)
　・Dress (DRESS)
　・Expression (FACE)

　Name the file following the rule `filename_LEVEL_DRESS_FACE`.

The character configuration is as follows:

"CharacterName" => %[
	"poses" => %[
		"PoseName" => %[
			"image" => "Composited character standing picture filename (filename_LEVEL_DRESS_FACE)",
			"dresses" => %[
				"DressVariationName" => "Dress variation filename (DRESS part)"
			],
			"defaultDress" => "Dress variation name to set as standard dress",
			"faces" => %[
				"ExpressionName" => "Expression filename (FACE part)",
			],
			"defaultFace" => "Standard expression filename's...",
		
                ],
            "defaultPose" => "Standard Pose Name",
            "voiceFile" => "Voice Filename",
            "voiceName" => "",
            ],

◇ Regarding Action Specification

It is possible to register layer movements in advance and
apply them asynchronously to layers.

(1) Registered Actions

Specify a set of registered parameters as an action.
The meaning of the parameters specified is the same as those in KAGEX.

A. Register as a dictionary in the `actions` item of envinit.tjs

envinit.tjs 
---------------------------------------------
    "actions" => %[
        "Bow" => %[
            "module" => LayerJumpOnceActionModule,
            "vibration" => -50,
            "cycle" => 3000,
            ],
    ...
    ],
---------------------------------------------

B. Create a script file named action_Name.tjs

action_Bow.tjs
---------------------------------------------
%[
  "module" => LayerJumpOnceActionModule,
   "vibration" => -50,
   "cycle" => 3000,
]
---------------------------------------------

In Scenario File
---------------------------------------------
[Minase action=Bow]

Or in shorthand notation

[Minase Bow]
---------------------------------------------

(2) When not registered in envinit.tjs

If the action name matches a KAGEX action extension module name,
it refers to other attributes for parameters and reproduces the action specification.
	
Description Example
--------------------------------------------------------------------
[Minase action=LayerJumpOnceActionModule vibration=-50 cycle=3000]
--------------------------------------------------------------------

☆ Precautions for Action Specification

Part 1

The following direct specifications are automatically replaced by the corresponding
commands in the world extension, so specifying them as actions has no meaning.

LayerFadeToModeModule  → opacity + time
LayerToRotateModule    → rotate  + time
LayerNormalZoomModule  → zoom    + time
LayerAccelMoveModule   → xpos, ypos, accel=1, time
LayerNormalMoveModule  → xpos, ypos, accel=0, time
LayerDeccelMoveModule  → xpos, ypos, accel=-1, time

Part 2

The "Result" of an action specification (such as when specifying time) is not saved as it looks visually.

・If execution was incomplete, it is reproduced from the beginning upon load.
・If execution was complete, it is ignored.

For example, if you use LayerNormalRotateModule with a time specification to stop at a specific angle
after a certain time, that angle is not fed back into the internal rotation angle,
and if you SAVE/LOAD in that state, the angle will be reset.
If you want to use it this way, please specify the target rotation angle separately using `rotate` etc.,
after the action specification.

Example:
// Rotate 1.5 times over a specified time using action, then stop
[event action=LayerNormalRotateModule angvel=180 time=3000 sync]
// Specify the position after completion (Even if rotation is skipped by click, it becomes this angle)
[event rotate=180]


◇ Regarding Transition Specification

(1) Registered Transitions

It is possible to specify a set of registered parameters as transition parameters.
The meaning of the parameters specified is the same as those in the KAG `trans` tag.

A. Register as a dictionary in the `transitions` item of envinit.tjs

envinit.tjs 
---------------------------------------------
    "transitions" => %[
        "NormalUpdate" => %[
            "time" => 1000,
            "method" => "crossfade",
            ],
---------------------------------------------

B. Create a script file named trans_Name.tjs

trans_NormalUpdate.tjs
---------------------------------------------
%[
  "time" => 1000,
  "method" => "crossfade",
]
---------------------------------------------

In Scenario File
---------------------------------------------
[Minase Uniform trans=NormalUpdate]

Or in shorthand notation

[Minase Uniform NormalUpdate]
---------------------------------------------

(2) When not registered in envinit.tjs

If the transition name matches a `trans` method name,
it refers to other attributes for parameters and reproduces the transition specification.
	
Description Example
--------------------------------------------------------------------
[Minase trans=crossfade time=1000]
--------------------------------------------------------------------

◇ Additional Parameters for Transitions

In transition definitions, besides those usable in KAG,
the following specifications are valid:

Attributes 	
	transwait
		Inserts a wait for the specified time after the transition.
	　　The wait is cancelled together with the transition cancellation.

	msgoff
		If the message window is displayed at the timing the transition is attempted,
		it hides the window before starting the process.
		
		※ This hiding process is the system's built-in fade-out.
		　 If you want to perform a specialized hiding process, please use the 
  		   msgoff command beforehand to hide it.

    charoff
        ※ Valid only for background images
		Erases character images when executing the transition.

	showaction
		Display-time action specification.
		If specified, this process is executed as an action when the character or layer is displayed.

	hideaction
		Hide-time action specification.
		If specified, this process is executed as an action when the character or layer is erased.

◇ Special Transitions

   ・ `method` is not described
　 ・ Both `showaction` and `hideaction` are described

　In these cases, instead of a transition as a standard Kirikiri feature,
　it becomes a new/old layer switching process using actions.

　・The old layer is erased by `hideaction`
　・The new layer is displayed by `showaction`

　At this time, the following additional parameters become valid:

　hideaction	Overwrites the time of hideaction (if omitted, same as time)
  showdelay		Delays the start time of showaction.
  hidedelay		Delays the start time of hideaction.

Adjust which one you want to happen first (showaction or hideaction)
using these parameters. By default, they happen simultaneously.

Example: Show first

 --------- Display Action (time)
 ......... hidedelay
          ----------- Erase Action (hidetime)

Example 2: Erase first
 ......... showdelay
           --------- Display Action (time)
 --------- Erase Action (hidetime)

　If nothing is specified, it waits for both actions to finish.
　If `nosync` is specified, it moves to the next process without waiting for the actions to finish.
　Using this, it is possible to specify different erasure methods for multiple characters.

● Creation of BGM

Basic Structure
-------------------------------
Intro Part
　　　　　　　← Label "start" ←-+
Loop Part                         |  Loop: Condition Flag 0 is 0
              ← Label "end"    --+
Coda Part
-------------------------------

☆ Use Kirikiri's Loop Tuner for labeling and loop specification.

Points during creation

(1) Do not abruptly cut off the end of the Coda part.
　　Insert music up to a good stopping point and apply a fade or similar.

　　→ This is so it can end cleanly when played once in sound mode, etc.
　　　 It is also considered acceptable to include one extra loop so that single playback loops twice before ending.

(2) The Intro part may be omitted.
　　Even in that case, include the label "start".

(3) Attach a flag to the loop. The condition is "Flag 0 is 0".

　　When playing BGM in KAG, if a loop exists,
　　it will loop forever regardless of whether the `loop` attribute is specified.
　　Define a flag for forced control of this.

・To play without an intro, specify "start" when playing.

● Regarding Track Gain Adjustment

・What can be done

　A function to adjust volume individually functionally during playback without modifying the data itself.
　Information goes into the ogg header, so there is no degradation.

・What tools are available

  VorbisGain 
  foobar2000

・Methodology

(1) Basically, create WAVs with volume adjusted first.
(2) If you absolutely need individual adjustment, you can adjust gain after creating OGG.

　Example: Only one BGM file is somehow too loud/quiet.
　　　Only specific voices are quiet, but re-ordering is a hassle, etc.
